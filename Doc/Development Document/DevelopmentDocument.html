Frontend

MemoryLib2.0 Frontend

Overview

The frontend has three basic components, namely the data organization panel (will be referred to as panel), task canvas (canvas), and chatbot agent (chatbot). The panel is responsible for data organization, visualization, editing, reviewing, and preparing raw materials for task canvas. The canvas serves as the functional interface that contributes to the final expected outcome of the entire memory experience. The chatbot is a core controller of the entire interface and serves as the most direct and low-cognitive-load port to interact with users. Users can prompt the chatbot to operate the canvas and panel. Comparing with the traditional daily task interface, we can make an analogue of current tools. For example, if we want to generate a multi-modality diary after a tour: the panel serves as raw data source, e.g. photo album, folder that stores raw data and notes, etc.; the canvas serves as the word doc or powerpoint that constitutes the diary; the chatbot serves as the AI agent that helps user summarize, manage, and draft the diary like ChatGPT or Gemini, with the only difference that they can directly understand raw memory pieces.

The elements of the three parts are shared, which means that the users can drag every element (e.g. an event, a piece of raw video/audio/photo/notes data, etc.) across the three components, even though they have different visual form across the three components.

Component 1

The Data Organization Panel (Panel)

The panel is responsible for initializing the memory layout, memory (event) editing, searching, CRUD(creating, reading, updating, and deleting). It serves as the raw material base for reviewing and operating more advanced tasks.

Cold Start

MemoryLib2.0 aims to collect massive personal data, therefore the initial visualization and organization is crucial since parallelly display every piece of information and event is impossible. Therefore we must apply filters, restrictions, and indices so that the information that relate to specific task is gathered and presented in a way that user will be easier to perceive. We apply a cold start procedure to lead users step by step to confirm the initial filters.

The frontend is responsible for collecting the range, time period granularity, purpose[a], indices of organization/filter. Specifically, range refers to the time range of the raw memory resource period (1-1); the time granularity refers to the different granularity of time. Currently, the minimal viable product could only include the granularity of daily record, in the future, it can support broader or more niche time period (1-2); purpose refers to the intention of usage of the raw memory data, e.g. memory review, completing certain tasks (diary, slides, reflection, etc.); indices refers to the backbone to organize the data, for example, if the data is going to be organized according to the time, perceived emotion, certain people, or certain core elements (2-1, 2-2, 3-1, 3-2). The design can be multi-layer, in the current minimal viable product requirements, we require at least two layers of indices (the detail of the usage of multi-layer indices organization will be elaborated in more details in X.XX). Users are required to fill in the primary index of data organization, while the second index can be default[b].

Initial Interface

After the cold start, the initial interface will be generated. The layout has two components, a filter/toolbar and the data organization panel. In the MVP, we can fix the layout (the event cannot be re-arranged by the user.

The top of the interface is a filtering panel, it has the components from the cold start stage (except purpose). The core components like primary index and secondary index will be listed directly on the interface, while some other more detailed adjustment will be triggered by the Settings icon[d]. After the confirmation by the user, the layout will be re-arranged according to the update. The detail of the editing, saving, will be in section X.XX.

In terms of the initial layout, it will be generated using fixed templates (only according to the primary index, the usage of other indices will be discussed in the following Zooming Function section). For example, if the primary index is semantic elements like event or emotion, the template will be a graph or bubble diagram like above. If the primary index is sequential such as time or advancement of certain event, the layout may be a coordinate-based like below.

Zooming Function

As mentioned earlier, as the memory data scale is massive, it is hard to review, manage, and edit in one interface, therefore, we applied a nested multi-layer structure to visualize memory data. In the MVP, the minimal product requires three layers of zoom-in and zoom-out. The first layer will be the initial interface discussed above. The second layer will be triggered when the user zoom in to check a certain element. Take the Event Type index for example, the initial interface will be a semantic graph. However, when the user for example want to check the detail of a specific keyword like Sports, they will be able to see the details of sports with more related events.
